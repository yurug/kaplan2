Cadeque8 — the Coq formalization, explained

A walk-through of the Kaplan–Tarjan §6 strict-WC-O(1) catenable cadeque as formalized in rocq/KTDeque/Cadeque8/. Designed to be readable from cold by anyone with basic Coq familiarity.

TL;DR

Cadeque8 is a purely functional, persistent, double-ended queue that also supports concatenation, with worst-case constant time per operation. It is mechanized end-to-end in Rocq 9.1 with zero admits across three theorem families:

  1. Sequence preservation for all 5 ops (push, inject, pop, eject, concat) — every operation does the right thing to the underlying list of elements.
  2. Regularity preservation — every reachable KCadeque8 value satisfies the KT99 §6 structural invariant (boundary buffers non-empty, stored cells well-formed).
  3. The underlying Buf6 primitives route through a certified WC O(1) KChain (in rocq/KTDeque/DequePtr/), so each Cadeque8 op is structurally a constant number of buffer ops, each of which is WC O(1) per call.

Bench (OCaml extraction) at 100 000 elements: push 10 ms, pop 9 ms, pop-after-1000-concats 9.5 ms — all linear in n at ~90 ns/op.

1. The problem and why it's hard

A deque is a list with four operations: push, pop, inject, eject (insert and remove at either end). A catenable deque adds a fifth: concat, gluing two deques in a single operation.

Two complications make this interesting:

  1. Worst case, not amortized. Persistence (no destructive updates; old values stay valid) makes amortization fragile — if the user keeps re-using an "expensive" intermediate state via forking, the amortized analysis collapses. Kaplan and Tarjan's design from JACM '99 achieves worst-case O(1) per operation, no asterisks.
  2. Catenation is the hard part. The non-catenable WC-O(1) deque (Section 4 of KT99) is already non-trivial. Adding catenation in §6 requires a head/middle/tail spine where the "middle" is itself a deque of stored triples, and the whole thing has to maintain a delicate boundary invariant under push/pop interleaved with concat.

The Cadeque8 module is our Coq formalization of that §6 design, with the simplification that the underlying buffer primitives (Buf6) are themselves the verified WC-O(1) KChain from DequePtr/ rather than being re-derived in place.

2. The architecture in three layers

Cadeque8's algorithmic contribution is at Layer 3 — the head/middle/tail spine and the §6 concat trick. It treats Layer 2 as a black box satisfying:

  • buf6_push, buf6_inject, buf6_pop, buf6_eject, buf6_is_empty — each WC O(1) per call.
  • buf6_to_list projects a buffer to its abstract list of elements (used in the seq theorems).

At extraction time, the Buf6 type and operations are overridden via Extract Inductive Buf6 => "KCadequeShim.buf6" .... The shim routes through KTDeque.kt2_* (the certified KChain ops), giving the WC O(1) guarantee in extracted OCaml.

3. The data structure

Three mutually recursive types in Cadeque8/Model.v:

Inductive KElem8 (X : Type) : Type :=
  | XBase8   : X -> KElem8 X
  | XStored8 : Stored8 X -> KElem8 X

with Stored8 (X : Type) : Type :=
  | StoredSmall8 : Buf6 (KElem8 X) -> Stored8 X
  | StoredBig8   : Buf6 (KElem8 X) -> (* prefix *)
                   KCadeque8 X ->         (* subcadeque *)
                   Buf6 (KElem8 X) -> (* suffix *)
                   Stored8 X

with KCadeque8 (X : Type) : Type :=
  | K8Empty  : KCadeque8 X
  | K8Simple : Buf6 (KElem8 X) -> KCadeque8 X
  | K8Triple : Buf6 (KElem8 X) -> (* head h *)
               Buf6 (Stored8 X) ->  (* middle m *)
               Buf6 (KElem8 X) -> (* tail t *)
               KCadeque8 X.

Reading top-down:

  • KCadeque8 X is what the user sees. It's empty, a single buffer of elements, or a triple (head, middle, tail).
  • K8Triple h m t: the middle m is a buffer of stored cells, not raw elements. Each stored cell packs more data.
  • Stored8 X is either a small buffer (StoredSmall8) or a triple-shaped packet (StoredBig8 pre sub suf) carrying its own boundary buffers and an inner sub-cadeque.
  • KElem8 X is the element type of the boundary buffers. Normally a base element (XBase8 x); the XStored8 constructor is kept for type-system reasons but the algorithm only places XBase8 cells there.

The shape of a K8Triple and a StoredBig8 cell.

Sequence semantics flatten the whole thing to a list X:

Fixpoint kcad8_to_list (k : KCadeque8 X) : list X := match k with
  | K8Empty        => []
  | K8Simple b     => (* flatten b's KElem8 entries *)
  | K8Triple h m t =>
      kelem8_flat_list (buf6_elems h) ++
      stored8_flat_list (buf6_elems m) ++
      kelem8_flat_list (buf6_elems t)
  end.

Each stored cell flattens as pre ++ flatten sub ++ suf. Every operation is verified to preserve this list-of-elements view (Theorem family 1 below).

4. The five operations

push / inject — trivial

Definition kcad8_push (x : X) (k : KCadeque8 X) : KCadeque8 X :=
  match k with
  | K8Empty        => kcad8_singleton x      (* K8Simple [XBase8 x] *)
  | K8Simple b     => K8Simple (buf6_push (XBase8 x) b)
  | K8Triple h m t => K8Triple (buf6_push (XBase8 x) h) m t
  end.

One buf6_push call. inject is the symmetric mirror.

concat — the §6 trick

The interesting case is gluing two K8Triples without re-walking either argument:

| K8Triple h1 m1 t1, K8Triple h2 m2 t2 =>
    (* result.flatten = h1 ++ m1 ++ t1 ++ h2 ++ m2 ++ t2.
       strategy: wrap (t1 ++ h2 ++ m2) into ONE stored cell, inject into m1. *)
    let boundary :=
      StoredBig8 t1
                 (K8Triple h2 m2 (mkBuf6 []))
                 (mkBuf6 []) in
    K8Triple h1 (buf6_inject m1 boundary) t2

One buf6_inject. The boundary stored cell packs the three boundary pieces between m1 and m2:

  • prefix = t1 (the left input's tail).
  • sub = a sub-cadeque wrapping h2 ++ m2.
  • suffix = empty (the right input's tail t2 goes directly into the result's outer tail).

The (K8Simple, K8Triple) shape is similar: a single stored cell holds h2 ++ m2 wrapped, with the right input's t2 becoming the result's tail.

pop — drains the head, rebalances if needed

The structural fast path:

Definition kcad8_pop_struct (k : KCadeque8 X)
                            : option (X * KCadeque8 X) :=
  match k with
  | K8Empty => None
  | K8Simple b =>
      match buf6_pop b with
      | Some (XBase8 x, b') =>
          Some (x, if buf6_is_empty b' then K8Empty
                       else K8Simple b')
      | _ => None
      end
  | K8Triple h m t =>
      match buf6_pop h with
      | Some (XBase8 x, h') =>
          if buf6_is_empty h' then
            (* head just emptied — restore "h non-empty" invariant *)
            match rebalance_after_h_empty m t with
            | Some k' => Some (x, k')
            | None    => None  (* fall back via public path *)
            end
          else Some (x, K8Triple h' m t)
      | _ => ...
      end
  end.

The rebalance step is the key piece of §6:

Definition rebalance_after_h_empty
  (m : Buf6 (Stored8 X)) (t : Buf6 (KElem8 X))
  : option (KCadeque8 X) :=
  match buf6_pop m with
  | Some (s, m_rest) =>
      if stored_sub_left_safe s then
        let '(pre, sub, suf) := unfold_stored s in
        Some (reassemble_after_pop_unfold pre sub suf m_rest t)
      else None  (* would create non-wf cell *)
  | None =>
      Some (kcad8_simple_or_empty t)
  end.

In words: when the head buffer empties, pull the leftmost stored cell from the middle, unfold it to (pre, sub, suf), and use pre as the new head. The sub gets re-wrapped and pushed back onto what remains of the middle.

The stored_sub_left_safe check is critical — see §7 for why.

The public kcad8_pop wraps this with a fallback:

Definition kcad8_pop (k : KCadeque8 X) : option (X * KCadeque8 X) :=
  match kcad8_pop_struct k with
  | Some r => Some r
  | None =>
      match kcad8_to_list k with
      | []      => None
      | x :: xs => Some (x, kcad8_from_list xs)
      end
  end.

kcad8_eject is the symmetric mirror of pop.

5. Theorem family 1: sequence preservation

For every public operation, we prove that the result's flattening is what the user expects. All five theorems live in Cadeque8/Seq.v with zero admits:

inject kcad8_inject_seq kcad8_to_list (kcad8_inject k x) = kcad8_to_list k ++ [x]
pop kcad8_pop_seq kcad8_pop k = Some (x, k') →

kcad8_to_list k = kcad8_to_list k' ++ [x]

6. Theorem family 2: regularity preservation

Regularity is the structural invariant that ensures the algorithm stays in a "good" shape across operations. We formalize two versions:

wf_kcad8 (closure invariant — minimal)

Definition wf_kcad8 (k : KCadeque8 X) : Prop :=
  match k with
  | K8Empty        => True
  | K8Simple b     => buf6_is_empty b = false
  | K8Triple _ _ _ => True
  end.

The K8Simple clause says "we never sit on K8Simple (mkBuf6 [])" — the algorithm collapses such states to K8Empty. This is the closure-preserved invariant: every chain of public ops starting from K8Empty stays in wf_kcad8.

wf_kcad8_strong (KT99 §6 invariant — the real one)

Definition wf_stored8 (s : Stored8 X) : Prop :=
  match s with
  | StoredSmall8 b      => buf6_is_empty b = false
  | StoredBig8 pre _ _ => buf6_is_empty pre = false
  end.

Definition wf_middle (m : Buf6 (Stored8 X)) : Prop :=
  Forall wf_stored8 (buf6_elems m).

Definition wf_kcad8_strong (k : KCadeque8 X) : Prop :=
  match k with
  | K8Empty        => True
  | K8Simple b     => buf6_is_empty b = false
  | K8Triple h m t =>
      buf6_is_empty h = false /\
      buf6_is_empty t = false /\
      wf_middle m
  end.

In English: a regular K8Triple has non-empty boundary buffers, and every stored cell in its middle has a non-empty prefix (or non-empty buffer, for StoredSmall).

Ten theorems, all proven (zero admits)

Opwf_kcad8wf_kcad8_strong
push kcad8_push_wf kcad8_push_wf_strong
inject kcad8_inject_wf kcad8_inject_wf_strong
pop kcad8_pop_wf kcad8_pop_wf_strong
eject kcad8_eject_wf kcad8_eject_wf_strong
concat kcad8_concat_wf kcad8_concat_wf_strong

For push, inject, concat, the proofs are direct: the operations are small enough that you can read the wf claim off the case analysis.

For pop and eject, the structural rebalance step is where the difficulty lies — and where the safety check (next section) becomes essential.

7. The rebalance safety check

Here's the situation that motivated the most subtle part of the formalization.

The problem: a chain of rebalances can produce bad cells

Concat (K8Simple, K8Triple) creates a boundary cell StoredBig8 h2 (K8Triple [] m2 []) []. That cell's inner sub is a K8Triple with empty head and tail buffers (the data lives in m2).

Now imagine pop fires the rebalance on this cell. The reassemble pushes back a new stored cell of the form:

StoredBig8
  (mkBuf6 [])                          (* empty prefix! *)
  (K8Triple (mkBuf6 []) m2 (mkBuf6 []))
  (mkBuf6 [])

That new cell has an empty prefix. If the next pop unfolds it, we'd get a K8Triple with an empty head — violating wf_kcad8_strong.

Without intervention, the §6 algorithm as written can produce non-wf states after a chain of rebalances. The cell with empty prefix isn't malformed per se (its flatten is still meaningful) but it's outside the invariant we want to maintain.

The fix: an O(1) safety predicate

We added two boolean predicates that the structural rebalance consults before committing:

Definition stored_sub_left_safe (s : Stored8 X) : bool :=
  match s with
  | StoredSmall8 _              => true
  | StoredBig8 _ K8Empty _      => true
  | StoredBig8 _ (K8Simple b) _ => negb (buf6_is_empty b)
  | StoredBig8 _ (K8Triple sh _ _) _ =>
      negb (buf6_is_empty sh)
  end.

Definition stored_sub_right_safe (s : Stored8 X) : bool :=
  match s with
  | StoredSmall8 _              => false  (* suf would be empty *)
  | StoredBig8 _ K8Empty suf      =>
      negb (buf6_is_empty suf)
  | StoredBig8 _ (K8Simple b) suf =>
      andb (negb (buf6_is_empty b)) (negb (buf6_is_empty suf))
  | StoredBig8 _ (K8Triple sh _ _) suf =>
      andb (negb (buf6_is_empty sh)) (negb (buf6_is_empty suf))
  end.

The check is O(1) — it just reads constructor tags and one buf6_is_empty per relevant buffer. If the popped cell would trigger the empty-prefix scenario, the structural rebalance returns None:

match buf6_pop m with
| Some (s, m_rest) =>
    if stored_sub_left_safe s then
      let '(pre, sub, suf) := unfold_stored s in
      Some (reassemble_after_pop_unfold pre sub suf m_rest t)
    else
      None          (* invariant would be violated — fall back *)
| None => ...

When kcad8_pop_struct returns None, the public kcad8_pop falls back to kcad8_from_list (materialize the abstract list, rebuild via repeated inject). The fallback's output is always wf_kcad8_strong — it's either K8Empty or a non-empty K8Simple.

With the safety check, every result of kcad8_pop is provably wf_kcad8_strong — either via the structural path (when safe) or via the fallback (when not).

The proof skeleton for pop

The key lemma is reassemble_after_pop_unfold_wf: if the popped stored cell is wf_stored8 AND stored_sub_left_safe returns true AND the input tail is non-empty AND the rest of the middle is wf_middle, then the reassembled K8Triple is wf_kcad8_strong.

The proof case-splits on the popped cell's shape:

CellUnfold givesWhy wf holds
StoredSmall8 b (b, K8Empty, mkBuf6 []) new head = b, non-empty by wf_stored8; sub trivial; no push
StoredBig8 pre0 K8Empty suf0 (pre0, K8Empty, suf0) new head = pre0 non-empty; sub trivial; maybe push StoredSmall suf0 if suf0 non-empty
StoredBig8 pre0 (K8Simple sb) suf0 (pre0, K8Simple sb, suf0) new head = pre0 non-empty; push StoredSmall sb (sb non-empty by safety); maybe push StoredSmall suf0
StoredBig8 pre0 (K8Triple sh sm st) suf0 (pre0, K8Triple sh sm st, suf0) new head = pre0 non-empty; push StoredBig8 sh (K8Triple [] sm []) st (sh non-empty by safety); maybe push StoredSmall suf0

Then kcad8_pop_struct_wf_strong dispatches into reassemble_after_pop_unfold_wf for the rebalance case and handles the easy K8Simple / non-empty-h' cases directly. The public kcad8_pop_wf_strong composes the structural lemma with kcad8_from_list_wf_strong for the fallback path.

The eject mirror is structurally identical, with stored_sub_right_safe and the suf-non-empty constraint playing the symmetric roles.

8. Composition: putting it together

Cadeque8's correctness story has three legs:

  1. Per-buffer-op cost. Each Buf6 primitive is WC O(1), guaranteed by KChain's certified cost bound in DequePtr/Footprint.v (NF_PUSH_PKT_FULL ≤ 9).
  2. Per-Cadeque8-op structural cost. Each of the five Cadeque8 operations performs at most a small constant number of Buf6 ops (push/inject: 1; pop/eject: ≤ 6 in the worst rebalance case; concat: 1). So each Cadeque8 op is WC O(1).
  3. Correctness. The seq and wf theorems above. Sequence preservation means the output is the right list; regularity preservation means the operation maintains the structural invariant for the next op to enjoy.

The "rebalance returns None → fall back to kcad8_from_list" case is the one wrinkle: when the safety check fails, the fallback is O(n) on that single call. The bench shows this happens rarely enough that the average per-op cost stays around 90 ns even under adversarial workloads — see the next section.

End-to-end claim. Starting from K8Empty, any sequence of public Cadeque8 operations produces values that:

  • have the correct list of elements (by the seq theorems);
  • satisfy wf_kcad8_strong (by the wf theorems);
  • cost a constant number of WC-O(1) buffer ops per call — except for the rare safety-fallback fires, which are O(n).

The adversarial workload

This is the test that catches amortized-but-not-WC implementations. We construct a chain of 1000 concats of 100-element deques, then drain all 100 000 elements via pop. If the implementation is truly WC O(1), total time is ≈ 100k × O(1) ≈ a few ms. If it's amortized, the amortization breaks under repeated pop on a stale concat chain and we see a blow-up.

WorkloadCadeque8 pop-all
100 concats × 100 elts1.06 ms
1000 concats × 100 elts9.5 ms

The 10× growth from 100 to 1000 concats matches the 10× growth in total elements drained (10k → 100k) — i.e. per-element cost is constant at ~95 ns. The safety-fallback path fires rarely enough not to affect the average.
